# AI System Transparency Statement

**Control**: AIUC-1 E017

**Purpose**: This document provides transparency into the design, capabilities, and limitations of the AI agent system used in the AIUC-1 SOC 2 Azure Foundry Compliance Lab. Its goal is to foster trust and understanding among users and stakeholders.

## 1. System Purpose

The system consists of a single AI agent (the SOC 2 Learning Agent) designed to perform Governance, Risk, and Compliance (GRC) assessments of live Azure infrastructure. The agent evaluates Azure resource configurations against the SOC 2 Trust Services Criteria, governed by the controls defined in the AIUC-1 standard.

This project serves as a portfolio piece to showcase the application of AI in a real-world compliance scenario.

## 2. AI Models & Data

| Component | Details |
| :--- | :--- |
| **Core Models** | The agent leverages gpt-4.1-mini from OpenAI, accessed via the Azure AI Foundry endpoint. No models are hosted within the project's Azure subscription. |
| **Orchestration** | Agent orchestration, tool execution, and state management are handled by the **Azure AI Foundry Agent Service**. |
| **Input Data** | The primary input data is the configuration of Azure resources, retrieved in real-time via custom Azure Functions. This data is supplemented by the content of the SOC 2 and AIUC-1 control documents. |
| **Output Data** | The system's output consists of compliance findings, governance documents, and infrastructure-as-code (IaC) suggestions. All outputs are stored as Markdown files or logs in the project's GitHub repository. |

## 3. Decision-Making Process

The system follows a **"tools provide data, agents provide judgment"** pattern.

1.  **Data Collection**: The SOC 2 Learning Agent uses Azure Functions as tools to fetch raw, objective configuration data from Azure APIs (e.g., "Is public blob access enabled on storage account X?").
2.  **Analysis & Judgment**: The agent then uses its understanding of the SOC 2 criteria to make compliance judgments (e.g., "Public blob access violates control CC5.").
3.  **Human-in-the-Loop**: For actions that modify the environment, a **mandatory human approval step** is enforced. The agent can generate and plan a change, but it cannot execute it without Pete's explicit approval.

## 4. Limitations & Known Issues

-   **No Real-Time Event Response**: The agent operates on an on-demand basis. It does not monitor for and respond to Azure events in real-time.
-   **Potential for Hallucination**: While measures are in place (groundedness checks, known-answer tests), the LLM may still hallucinate or generate factually incorrect information. All findings should be independently verified before being considered definitive.
-   **Scope-Bound**: The agent can only assess what it is instructed to look at. A misconfiguration in a resource not included in the target list will not be detected.
-   **No Self-Improvement**: The agent does not learn from experience or automatically update its own instructions. All improvements are manually implemented by Pete.
-   **Dependency on External APIs**: The system's functionality is entirely dependent on the availability and performance of the Azure AI Foundry and OpenAI APIs.
